基于日线的 A 股量化交易模拟（一年）
## 题目背景

**本题输入量较大，请使用较快的读入方式。**

你需要实现一个交易算法与交互程序。Judge（交互端）会按日向你的程序发送 $200$ 支 A 股在每个交易日的行情数据；你在收到当天（或上一日）行情后可以提交买卖委托，这些委托将在**下一交易日开盘价**被 Judge 逐条执行（最后一个交易日允许在**当日收盘价**执行一次提交）。初始现金为 $m = 10^5$ 元人民币。最终得分为一年交易结束后的**总资产**（现金 + 持仓市值），得分越高越好。

请注意该模型是极度简化的真实 A 股模型，不构成投资建议，若在真实市场交易中使用您编写的代码，后果自负。

## 题目描述

* 设股票池大小为 $n=200$。交易日总数为 $D$（Judge 在初始化时提供），交易日编号从 $1$ 到 $D$。
  * 初始现金为 $m = 10^5$（单位：人民币元）。
* 交易最小单位（1 手）为 $L = 100$ 股。
* 手续费率（佣金）为 $\alpha$（浮点，Judge 在初始化时给出）；每笔最低佣金为 $\mathrm{com_{\min}}$（元）。
* 卖出时需缴纳印花税率 $\beta$。
* 单日委托条数上限为 $K_{\max}$。
* Judge 会在初始化阶段把这些参数、以及被选中的 $200$ 支股票及基准日（day 0）行情发送给你的程序。
* 每天 Judge 向你的程序发送当日 $n$ 支股票的行情记录，字段如下（顺序固定）：

```
["date","code","open","high","low","close","volume","amount","amplitude","pct_chg","change","turnover"]
```

各字段含义与格式见下节。

你的程序在收到 day $t-1$ 的行情后（或初始化的 day 0）可提交 $0$ 或多条委托（每条 `BUY code qty` 或 `SELL code qty`），并在提交完毕后输出一行 `DONE`，随后 flush（参见“交互流程”）。Judge 会在 **day t 的开盘价**（即 day t 提供的 `open` 值）逐条执行你提交的委托，并在执行完后把 day t 的全部行情行发送给你；如此循环直到 day $D$。在收到 day $D$ 的行情后，你仍可提交委托；Judge 规定：**你在收到 day $D$ 行情后提交的委托将在 day $D$ 的收盘价 `close` 执行**（这是最后一次执行）。交互结束后，Judge 输出最终资产并结束会话。

## 选股规则（Judge 的实现细则）

* Judge 在题目准备阶段从上交所与深交所中选取样本股：

  1. 剔除 ST/*ST/退市风险等特殊股及停牌时间过长的标的；
  2. 以 $5$ 种方式取 $200$ 支，对应 $5$ 组不同的数据。具体的选股方式不公开。

## 行情数据字段（每只股票每日记录）

每行由 $12$ 个字段组成，字段顺序固定且 Judge 会保证格式统一（逗号或空格分隔，会在题内说明，参赛时以 Judge 文档为准）：

1. `date` — 交易日，格式 `YYYY-MM-DD`（字符串）。
2. `code` — 股票代码（字符串，例如 `"600000"`）。
3. `open` — 当日开盘价（元/股，浮点，保留两位小数）。
4. `high` — 当日最高价（元/股）。
5. `low` — 当日最低价（元/股）。
6. `close` — 当日收盘价（元/股）。
7. `volume` — 成交量（股，整数）。
8. `amount` — 成交额（元）。
9. `amplitude` — 振幅（百分数，例如 `5.10` 表示 $5.10\%$）。
10. `pct_chg` — 涨跌幅（相对前日收盘，百分数）。
11. `change` — 涨跌额（元/股）。
12. `turnover` — 换手率（百分数）。

> Judge 保证：所有价格字段均为非负且具有两位小数；停牌时可能用 `open = 0` 或特定 `NA` 标识（Judge 会在输入中使用同一约定）。成交量为整数。所有日期为真实交易日（无周末或节假日混入）。

## 交互协议（严格顺序与格式）

**注意：交互是严格顺序的，参赛程序必须按下面流程与 Judge 对话；每次输出 `DONE` 后务必调用 `fflush(stdout)` 或等价操作以确保数据发送到 Judge。**

### 1. 初始化（Judge → 参赛程序）

Judge 首先发送一行初始化参数（空格分隔）：

```
INIT n D m L alpha com_min beta K_max
```

示例：

```
INIT 200 244 100000 100 0.0003 5 0.001 500
```

随后 Judge 发送 $n$ 行基准日（day 0）行情（每行为一只股票的 $12$ 个字段，字段分隔符参见 Judge 的 IO 细则）。在 Judge 发送完这 $n$ 行并且 flush 后，参赛程序可以基于 day 0 的行情提交用于 day 1 开盘执行的委托。

### 2. 每日循环（对 $t = 1..D$）

流程如下（精确步骤）：

1. **起点**：你已经收到了 Judge 发来的 day $t-1$ 的 $n$ 行行情（对于 $t=1$，即已收到 day 0）。

2. **你 → Judge**：输出 $0$ 到 若干 条委托，每条一行，格式严格为：

   ```
   BUY <code> <qty>
   SELL <code> <qty>
   ```

   * `<code>` 必须是 Judge 初始化时提供的股票代码之一。
   * `<qty>` 为买/卖数量（股），必须为正整数，且必须是 $L$ 的倍数（例如 $100$、$200$ ……）。
   * 允许对同一只股票多次下单；Judge 将按提交顺序依次处理。
   * 提交完委托后必须输出单独一行：

     ```
     DONE
     ```

     输出 `DONE` 后 **立即** 调用 `fflush(stdout)`（或你所用语言的等价函数），然后等待 Judge 的响应（不要继续读/写直到接收到 Judge 的下一段输入）。

   约束：单次提交的委托行数不得超过 $K_{\max}$。若超过，Judge 会返回 `INVALID` 或 `ERROR` 并中止交互（得分 $0$）。

3. **Judge → 你**：Judge 在 **day t 的开盘价**（即 day t 的 `open` 字段）逐条按你提交的顺序尝试执行委托（详见“委托执行规则”）。执行结束后（或在遇到停牌等失败情况并记录后），Judge 将把 day t 的全部 n 行行情发送给你（这时你已经完成了 day t 的开盘执行并且持仓/现金被更新）。接着进入下一轮：你可基于 day t 的行情提交用于 day $t+1$ 的委托。

4. **当 $t = D$ 特殊说明**：在收到 day $D$ 的行情后，你仍然可以提交委托（格式同上）。Judge 在收到你最后一次 `DONE` 后 **将在 day $D$ 的收盘价 `close` 执行这些委托（而不是下一日开盘）**。执行并结算后 Judge 输出 `FINISH <FinalAsset>` 并结束会话。

### 3. 交互命令集合（完整）

* `BUY code qty` — 买入 `qty` 股（must be multiple of `L`）。
* `SELL code qty` — 卖出 `qty` 股（must be multiple of `L`，且不得超过当前持仓）。
* `DONE` — 表示本轮委托提交结束并 flush（必须）。

任何其它未定义命令都视为非法，Judge 将返回 `INVALID` 并中止会话。

## 委托执行规则（细则）

1. **执行时间点与价格**：

   * 一般情况下，所有在收到 day $t-1$ 行情后提交并被 Judge 接受的委托，会在 day $t$ 的 `open` 价格处尝试执行（市价成交，成交价 = `open`）。
   * 在最后一日（$t = D$），你在收到 day D 行情后提交的委托将以 day D 的 `close` 价格成交（市价成交，成交价 = `close`）。
   * 若执行日该股票停牌（`open==0` 或 Judge 的停牌标记），该笔委托**不成交**且直接丢弃（不会自动重试或排队）。Judge 会在日志/返回中说明该笔委托失败原因。
   * 为更贴近真实市场的撮合行为，Judge 会在执行时对 **买入委托** 应用涨停判断并 **拒绝在涨停日买入**。

2. **买入（BUY）结算**：

   * 成交额（不含佣金）：$V = p \cdot q$，其中 $p$ 是执行价（元/股）， $q$ 是委托股数（股）。
   * 佣金：$\mathrm{commission}(V) = \max(\mathrm{com\_min},\ \alpha\cdot V)$（按 Judge 初始化给定的 $\alpha$ 与 $\mathrm{com\_min}$ 计算）。
   * 买入导致现金减少：
     $$
     \Delta \text{cash} = V + \mathrm{commission}(V).
     $$
   * 若当前现金不足以支付 $\Delta \text{cash}$，该委托被**拒绝（不部分成交）**，Judge 跳过该委托并继续处理下一个委托。

3. **卖出（SELL）结算**：

   * 成交额：$V = p \cdot q$。
   * 印花税：$\beta \cdot V$（仅卖出适用，按 Judge 给定 $\beta$）。
   * 佣金：$\mathrm{commission}(V) = \max(\mathrm{com\_min},\ \alpha\cdot V)$。
   * 卖出导致现金增加：
     $$
     \Delta \text{cash} = V - \beta\cdot V - \mathrm{commission}(V).
     $$
   * 若当前对该股票持仓不足 $q$ 股，则该委托被拒绝（不允许卖空），Judge 跳过该委托并继续处理下一个委托。

4. **委托顺序与状态更新**：

   * 在同一执行时刻（同一开盘或最后日收盘）有多笔委托时，Judge 按你提交的顺序逐条执行，并在每笔委托后即时更新现金与持仓；后续委托按更新后的账户状态判断能否执行（例如前一笔买单占用了现金会使下一笔买单因现金不足而被拒绝）。

5. **四舍五入规则**：

   * Judge 在每一步计算（成交额 $V$、佣金、印花税、现金余额）均以人民币的“分”为最小单位进行四舍五入（即保留两位小数，标准四舍五入）。参赛者在本地模拟或调试时应使用相同规则以确保结果一致。

## 说明/提示

## 约束与非法情况

* 委托 `qty` 必须为 $L$ 的倍数（否则 Judge 返回 `INVALID` 并中止）。
* 不允许卖空（持仓不得为负）；不支持融资融券。
* 单次提交委托行数不得超过 $K_{\max}$。超过上限 Judge 将返回 `INVALID` 并中止。
* 委托必须使用 Judge 初始化时提供的代码；非法股票代码视为格式错误。
* 任意格式不符或未定义命令会导致 Judge 返回 `INVALID` 并中止交互（对应提交得分为 $0$）。

## 最终得分（评分方式）

Judge 在交互结束（执行完最后一笔委托）后计算最终资产（FinalAsset）：
$$
\text{FinalAsset} = \text{cash}_{\text{end}} + \sum_{i=1}^{n} h_i \cdot \text{close}_{D}(i),
$$
其中：

* $\text{cash}_{\text{end}}$ 为交互结束时的现金余额（保留两位小数）。
* $h_i$ 为你持有的第 $i$ 支股票的股数（股）。
* $\text{close}_{D}(i)$ 为第 $i$ 支股票在最后交易日 $D$ 的收盘价（Judge 在 day $D$ 行情中提供）。

Judge 最终输出：

```
FINISH <FinalAsset>
```

例如：

```
FINISH 123456.78
```

该 `FinalAsset` 为你的得分（浮点数，保留两位小数）。在比赛中，提交按得分从高到低排名；如需平局决胜，主办方可追加其它评价指标（例如最大回撤、夏普比率或提交时间），在本题默认不指定额外 tie-break。

## 提示与实现建议（对参赛者）

* **务必处理 I/O 的 flush**：每次输出 `DONE` 后必须 flush；否则 Judge 会一直等待输入或发生超时。下文给出常见语言的 flush 示例代码片段（示例仅作参考）：

  * **C++**

    ```cpp
    cout << "BUY 600000 100\n";
    cout << "DONE\n";
    cout.flush(); // 或 `fflush(stdout);`
    ```

  * **Python**

    ```python
    print("BUY 600000 100")
    print("DONE", flush=True)
    # 或
    sys.stdout.write("DONE\n")
    sys.stdout.flush()
    ```

  * **Java**

    ```java
    System.out.println("BUY 600000 100");
    System.out.println("DONE");
    System.out.flush();
    ```

* **严格按照四舍五入规则计算费用与余额**：所有金额应以分为单位四舍五入到两位小数；在本地回测时使用相同规则以避免与 Judge 不一致。

* **处理拒单与失败情况**：Judge 会跳过因资金不足或持仓不足导致的被拒委托。你的策略应考虑委托顺序对后续委托的影响。

* **停牌/复牌处理**：若执行日 `open` 为 `0` 或标注停牌，委托失败并被丢弃。不要假设委托会被自动排队或延迟执行。

* **不要假设日内成交量分配或部分成交**：所有委托要么全部成交（满足现金/持仓约束），要么被拒绝（不部分成交）。

* **遵守每次提交委托行数上限 $K_{\max}$**。

## 错误/异常返回（Judge 行为规范）

* 若 Judge 检测到网络/通信异常或参赛程序长时间无输出，Judge 可判定超时（同样为失败）。
